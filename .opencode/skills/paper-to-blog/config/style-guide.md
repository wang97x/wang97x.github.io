# Writing Style Guide

论文博客写作风格指南 - 从 blog-writer skill 迁移

## 段落组织

### 段落长度
- **推荐**: 2-4 句话 per 段落
- **避免**: 超过 5 句的长段落（考虑拆分）
- **例外**: "一句话总结"部分可以是单个长句

### 小标题频率
- **推荐**: 每 2-3 段一个 `###` 标题
- **目的**: 帮助读者快速浏览和定位
- **格式**: 使用 `###`（三级标题），避免 `##`（二级标题已在模板中定义）

### 字数范围
- **总计**: 2000-3000 字
- **各部分建议**:
  - 一句话总结：50-100 字
  - 背景与动机：300-500 字
  - 核心方法：800-1200 字
  - 实验结果：400-600 字
  - AI 分析方法亮点：300-500 字
  - 总结：200-300 字

## 语气和语调

### 核心原则
- **直接**: 陈述观点明确，不模棱两可
  - ✅ "这个方法无效"
  - ❌ "这个方法可能不是最优的"
  
- **观点鲜明**: 允许表达 contrarian 观点
  - ✅ "这篇综述的 4W 分类法是过度设计"
  - ❌ "有些人可能认为这个分类法有用"

- **第一人称**: 分享经验时使用"我"
  - ✅ "我发现这个技巧很有效"
  - ❌ "人们发现这个技巧很有效"

### 过渡词
**推荐**:
- that said
- however
- in contrast
- building on this
- crucially

**避免**:
- furthermore（过于正式）
- moreover（过于正式）
- additionally（冗余）

## 词汇选择

### 推荐词汇
| 推荐 | 避免 | 示例 |
|------|------|------|
| leverage | utilize | "leverage pre-trained models" |
| key | important | "key insight" |
| show | demonstrate | "results show that" |
| use | employ | "use attention mechanism" |
| help | facilitate | "help reduce latency" |
| need | require | "we need to address" |

### 技术术语
- **首次出现时解释**: "思维链（Chain-of-Thought, CoT）"
- **保持一致**: 全文使用同一术语（不要混用 CoT 和思维链）
- **避免过度缩写**: 除非是领域标准（如 LLM, GNN）

## 列表使用

### 优先使用段落
✅ **推荐**（连贯叙述）:
```markdown
这篇论文的核心贡献是提出了自洽思维链蒸馏方法。通过在教师端生成多条推理路径，学生端学习自洽的子集，该方法在保持推理质量的同时显著降低了计算开销。
```

❌ **避免**（过度列表化）:
```markdown
这篇论文的贡献包括：
- 提出了自洽思维链蒸馏方法
- 在教师端生成多条推理路径
- 学生端学习自洽的子集
- 降低了计算开销
```

### 适合列表的场景
- **对比**: 方法 A vs 方法 B
- **总结**: 3-5 个关键发现
- **步骤**: 清晰的流程（1, 2, 3...）

### 表格使用
适合对比多个方法/维度：
```markdown
| 方法 | 准确率 | 速度 | 参数量 |
|------|--------|------|--------|
| 方法 A | 85% | 快 | 小 |
| 方法 B | 87% | 中 | 中 |
| Ours | 89% | 快 | 小 |
```

## 公式和代码

### 数学公式
- **独立公式**: 使用 `$$...$$`
  ```markdown
  $$G(t_i|a^*)=\log\frac{P(t_i|p,q,a^*,t_{<i})}{P(t_i|p,q,a^{'},t_{<i})}$$
  ```

- **行内公式**: 使用 `$...$`
  ```markdown
  注意力权重 $\alpha_i$ 的计算...
  ```

### 代码片段
- **短代码**: 行内 `` `code` ``
- **长代码**: 使用代码块
  ````markdown
  ```python
  def forward(self, x):
      return self.attention(x)
  ```
  ````

## 图片说明

### 格式要求
```markdown
*图 X：说明文字，解释图片展示的核心概念*
```

### 好的说明
✅ *图 2:4W 记忆分类法，包含四个正交维度：When（生命周期）、What（记忆类型）、How（存储形式）、Which（模态类型），每个维度下又有细分子类别*

### 差的说明
❌ *图 2:4W 分类法*（太简短，未解释内容）

### 说明要素
1. **是什么**: 图片类型（框架图、结果图、表格等）
2. **展示什么**: 核心概念或关系
3. **为什么重要**: 与上下文的关联（可选）

## "AI 分析方法亮点"写作指南

### 三个维度

#### 1. 问题定位精准
**写作要点**:
- 直接针对论文解决的核心问题
- 说明为什么这个问题重要
- 指出之前方法的不足

**示例**:
> 这篇综述直击 AI 记忆研究的核心痛点：概念碎片化。当前论文中"记忆"一词被滥用到几乎失去意义——有人把 RAG 叫记忆，有人把向量数据库叫记忆。通过明确区分 LLM Memory、Agent Memory、AI Memory 三层概念，论文为后续讨论建立了清晰的边界。

#### 2. 方法创新
**写作要点**:
- 技术/方法上的创新点
- 包含具体数据（提升百分比、参数量等）
- 与基线方法对比

**示例**:
> 本文提出了 4W 记忆分类法，用四个正交维度（When/What/How/Which）统一定义记忆机制。关键创新在于可操作性——When 帮你决定 TTL，What 帮你选择存储格式，How 帮你权衡效率，Which 帮你规划多模态融合。这比单纯的"短期/长期"二分法更能捕捉记忆系统的复杂性。

#### 3. 实用性强
**写作要点**:
- 实际应用价值或优势
- 包含性能对比（延迟、准确率等）
- 给出使用建议或场景

**示例**:
> 论文梳理了 10+ 应用场景，从对话助手到具身机器人。关键发现：图结构记忆如果设计不当（如 Zep），存储开销可能超过原始上下文的 20 倍！这为工程实践提供了重要警示。建议优先尝试分层记忆架构 + 向量数据库的组合。

## 参考链接格式

```markdown
## 参考链接

1. 论文原文：[arXiv:2305.01879](https://arxiv.org/abs/2305.01879)
2. 代码仓库：[GitHub](https://github.com/user/repo)
3. 项目主页：[链接](https://example.com)
4. 相关工具：[工具名](https://tool.com)
```

**要求**:
- ✅ 只保留公开的链接
- ✅ 使用简短描述（不要直接贴 URL）
- ✅ 按重要性排序（论文 > 代码 > 主页 > 其他）

## 常见错误

### 学术化表述
❌ "本文提出了一种新颖的方法..."
✅ "这篇论文提出了..."

### 过度谦虚
❌ "我们初步实验表明..."
✅ "实验结果显示..."

### 模糊描述
❌ "效果很好"
✅ "在 CSQA 基准上提升 3.2%"

### 企业术语
❌ "赋能"、"抓手"、"闭环"
✅ "支持"、"方法"、"流程"
