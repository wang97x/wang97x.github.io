## 一句话总结

`Loop Engineering` 讲的是如何把 Agent 工作流做成一个长期可运行、可验证、可审计的闭环系统；`/goal` 则是这个思路在单个 coding session 里的最小可行版本。前者偏“建系统”，后者偏“先把 loop 跑起来”。

---

## 为什么现在要聊 Loop，而不是只聊 Prompt

过去很多人使用 AI 编程，停留在“写一个 prompt，让模型给一段代码”的阶段。  
但一旦任务变成持续性的工程协作，比如修复测试、整理 PR、处理 CI、分步骤迁移模块，真正决定效果的往往不再是某一句 prompt，而是：

- 任务能不能反复执行
- 结果能不能自动验证
- 失败之后能不能重试或恢复
- 运行过程能不能被人检查
- 多个 agent 会不会互相干扰

这时你需要的就不是一次性提示词，而是一个可以持续推进目标的 **loop**。

从这个角度看，所谓 `Loop Engineering`，本质上是在回答一个问题：

> 如何把“模型会做事”升级成“系统能稳定做成事”？

---

## 什么是 Loop Engineering

如果把 Agent 看成一个会写、会改、会调用工具的执行者，那么 `Loop Engineering` 关注的是它所处的整个运行环境。

从这份笔记里，可以把它理解为三个层面。

### 第零层：先判断这件事值不值得做成 loop

不是所有任务都适合 loop。  
真正适合的任务通常同时满足下面几个条件：

- **有重复性**：不是只做一次，而是会反复出现
- **能自动验证**：有测试、类型检查、lint、build 这类硬门槛
- **成本扛得住**：token 和反复重试的开销在收益之内
- **过程可复现**：有日志、有环境、有明确的输入输出
- **有人愿意 review**：最终产出不是无条件直推

如果连最基本的验证和复查都没有，那它更像一场“黑箱生成”，而不是一个可靠 loop。

所以第一步不是造系统，而是先筛任务。

适合作为好种子的任务通常是：

- CI 失败分类
- 依赖升级
- lint-and-fix
- 小范围 issue 到 PR 草稿

而不适合第一批上 loop 的，往往是：

- 大规模架构重写
- 支付、鉴权、部署这类高风险链路
- 目标本身就很模糊的产品判断类工作

### 第一层：一个能跑的 loop 至少要有哪几块

这份笔记把 loop 拆成了五个核心构件，这个抽象非常实用：

| 构件 | 作用 | 你要解决的问题 |
|---|---|---|
| Automation | 负责心跳、调度、停止条件 | loop 什么时候启动，什么时候该停 |
| Worktree | 负责隔离 | 并行 agent 会不会互相踩代码 |
| Skill | 负责项目记忆 | agent 是否知道目录职责、禁区、验证命令 |
| Connector / MCP | 负责接外部工具链 | 能不能和 GitHub、工单、告警系统协同 |
| Sub-agents | 负责 maker/checker 分离 | 写代码的人和验收的人是不是同一个“脑子” |

这五块拼起来，才是一个比较完整的 Agent 闭环环境。

### 第二层：最小可行 loop 的搭建顺序

很多人一上来就想做多 agent、自动调度、接 GitHub、接 Slack，但这通常会把复杂度拉爆。  
更稳的顺序应该是：

1. 先手动跑通一次任务
2. 把项目规则沉淀成 `SKILL.md` 或等价说明
3. 增加 `STATE.md` 之类的进度记录
4. 补上硬 gate：测试、lint、typecheck、build
5. 再把它包成自动化 loop
6. 真的需要并行时，再引入 worktree
7. 真的需要进入真实工作流时，再接 connector

这条顺序背后的核心纪律是：

> 先让一次手工执行稳定，再自动化；先保证会停、会验，再追求更快。

### 第三层：系统上线后最难的是守纪律

一个 loop 能不能长期可用，最终取决于它能否避开几类常见风险：

- **假装完成**：agent 自己宣布 done，但没有真正过 gate
- **理解债务**：代码改动越来越多，但团队越来越看不懂
- **安全扩大化**：无人值守的 loop 逐渐获得过大的权限

所以真正成熟的 loop，需要的不只是“会跑”，还要有：

- 独立 checker
- 硬验证门槛
- 定期人工抽查
- 权限回收和安全扫描

这也是为什么很多团队最后发现：难点不在让 agent 干活，而在让这个系统长期不失控。

## /goal 是什么，它和 Loop Engineering 的关系是什么

这份笔记里最有价值的一句话，我觉得就是：

> `/goal` 不是 Loop Engineering 的全部，它是 Loop Engineering 在单个 coding session 里的 MVP。

这句话几乎把两者的关系讲透了。

### /goal 解决的是“今天就能跑起来”

`/goal` 的价值，不在于它把完整系统一次性搭好，而在于它把 loop 的核心骨架压缩成了一个你今天就能用起来的交互入口。

你只需要给它：

- 一个清晰的目标
- 一套完成证据
- 一组限制条件
- 一个停止条件

然后 agent 就会在当前会话里围绕这个目标持续推进：写代码、验证、修复、再验证，直到完成或被你暂停。

换句话说，`/goal` 把 loop 这件事从“系统工程”降维成了“工作方式升级”。

### 从维度上看，两者差别在哪里

如果把两者放在一起比较，会更清楚：

| 维度 | Loop Engineering | /goal |
|---|---|---|
| 本质 | 一个完整的 Agent 闭环系统 | 一次会话中的最小闭环控制器 |
| 生命周期 | 可跨会话、跨重启、长期运行 | 主要绑定当前 session |
| 状态管理 | 倾向于显式状态文件或外部系统 | 更偏当前线程中的隐式状态 |
| 并行能力 | worktree、多 agent、队列都可接入 | 通常单目标串行推进 |
| 外部协作 | 适合接 GitHub、Jira、Slack、Sentry | 以本地仓库和本地验证为主 |
| 安全与审计 | 强调 gate、权限、审计、review | 更轻量，也更依赖操作者自觉 |
| 上手成本 | 高 | 低 |
| 适用阶段 | 团队级、常态化、可运营场景 | 个人或小团队的起步阶段 |

如果把 `Loop Engineering` 比作一套工厂产线，那么 `/goal` 更像是你先在工作台上搭起来的一套小型流水线。

它不是最终形态，但它让你先体验到闭环的手感。

## 为什么 /goal 是理解 Loop Engineering 的最好入口

很多方法论最大的问题，是概念很完整，但落地门槛太高。  
你知道“要有状态、要有验证、要有停止条件、要有人类 review”，但你很难一上来就把这些全部搭出来。

`/goal` 的意义就在这里。

它把 loop 这件事最关键的几个要素，压缩成了一个足够小的实践单元：

- **目标**：你到底要 agent 完成什么
- **证据**：怎么证明它真的做完了
- **约束**：哪些目录能动，哪些不能动
- **停止条件**：卡住时什么时候暂停，而不是无限猜

只要你开始认真写 `/goal`，你的思维就会被迫从“给模型一个愿望”切换到“给系统一份验收合同”。

这其实正是 `Loop Engineering` 想训练出来的能力。

## /goal 真正的关键，不是命令本身，而是目标写法

很多人第一次用 `/goal`，容易把它当成“更长一点的 prompt”。  
但从笔记里的经验看，真正的分水岭在于：**目标是否可验证。**

### 坏目标长什么样

下面这类目标看起来合理，但对 agent 来说约束太弱：

- 优化一下性能
- 完善 auth 模块
- 把代码整理得更干净

问题不是它们不重要，而是它们没有清晰终点。

### 好目标长什么样

一个好的 `/goal`，应该至少同时写清：

- **目标**：一句话说清终点
- **完成证据**：用测试、benchmark、命令输出来证明
- **约束**：不准动什么，必须兼容什么
- **范围**：只准在哪些目录内行动
- **停止条件**：卡住时如何上报，而不是盲猜

例如：

```text
/goal
目标：将 src/auth 从 callback 迁移到 async/await
完成证据：npm test -- --testPathPattern=auth 全绿，且 src/auth 中不再出现 callback
约束：不修改 tests 文件，不改变外部 API 签名
范围：src/auth/
停止条件：连续 3 次在同一错误上失败后暂停，并报告阻塞点
```

这样的目标之所以强，不是因为字多，而是因为它把“完成”定义成了可以验收的事实。

## 怎么判断一个团队该停留在 /goal，还是升级到完整 loop

这其实不是“工具偏好”问题，而是“组织规模和任务形态”问题。

### 先停留在 /goal 的场景

如果你现在处在下面这些阶段，用 `/goal` 往往是更好的选择：

- 刚开始尝试让 agent 连续完成任务
- 团队还没有稳定的 gate 和 review 流程
- 任务范围小、验证清晰，但不值得重投入系统建设
- 想先验证“闭环方式”是否真的省时间

简单说：  
你还在证明“loop 值不值得”，那就先别过早系统化。

### 该升级到 Loop Engineering 的场景

如果已经出现下面这些信号，就说明该往完整 loop 系统走了：

- 同类任务开始高频重复出现
- 多个 agent 需要并行处理不同目标
- 本地会话已经不够，需要跨天、跨人、跨系统持续执行
- 需要接工单、PR、告警、审计链路
- 无法再依赖人工记忆来保证规则一致性

这时 `Loop Engineering` 的价值才真正显现出来，因为你已经不是在“尝试”，而是在“运营”一套 agent 工作流。

## 一条很实用的递进路线

如果让我把这两篇笔记收束成一条最实操的建议，我会推荐下面这条三段式路线。

### 第一步：先用 /goal 跑一个小闭环

找一个风险低但验证明确的任务，比如：

- 修掉一组 lint 错误
- 修复单个模块测试
- 做一次小范围依赖升级

重点不是任务本身，而是你要亲手体验一遍：

> 目标 -> 迭代 -> 验证 -> 重试 -> 停止

只要这个闭环跑通，你对 loop 的理解就会立刻从抽象变具体。

### 第二步：把经验沉淀成规则和状态

当你开始反复使用这类闭环，就别再依赖临场口头说明了，而是逐步补齐：

- 项目规则文档
- 验证命令清单
- `STATE.md` 进度记录
- 禁区和边界条件

这一步的本质是把“你脑子里的经验”外化成 agent 可读、可执行的环境。

### 第三步：只有在收益成立时，再升级成系统

等你确认闭环真的能省时间、提高吞吐、降低重复劳动，再去做：

- worktree 并行
- CI / cron 调度
- 工具链 connector
- maker/checker 分离
- 更正式的审计与安全 gate

这时你的升级才是顺势而为，而不是为了“看起来先进”而堆复杂度。

## 这两篇笔记真正讲的是同一件事

把两篇内容合起来看，会发现它们讨论的其实不是两套独立方法，而是同一个主题在两个尺度上的展开：

- `Loop Engineering` 讨论的是：如何设计一套长期可靠的 Agent 闭环系统
- `/goal` 讨论的是：如何在当前终端会话里，把闭环思维压缩成一个最小可执行单元

一个偏系统设计，一个偏即时操作；一个回答“如何建”，一个回答“如何先跑起来”。

所以与其把它们看成替代关系，不如把它们看成递进关系：

> `/goal` 是你进入 Loop Engineering 思维的最短路径。  
> Loop Engineering 则是你把这种思维从个人技巧升级成团队系统的方法。

---

## 总结

如果只记一件事，我觉得应该记住这句：

> prompt 只是一次指令，loop 才是持续交付的结构。

`/goal` 的意义，在于让你用很低的成本，先体验这种结构；而 `Loop Engineering` 的意义，在于当这种结构被证明有价值后，进一步把它做成可重复、可扩展、可审计的工程系统。

对大多数团队来说，最好的路径往往不是一开始就追求“完整自治”，而是先用 `/goal` 跑通一个可验证的小闭环，再决定是否值得往完整 loop 系统升级。
